# DocMee `/docs-v2` 独立文档入口设计

## 背景
当前文档站已经覆盖 `UI 接入`、`快速开始`、`API 参考`、`开放能力` 和 `定价` 等主要内容，但入口层仍然偏向开发者目录式阅读。用户希望新增一个独立的 `/docs-v2` 入口，先帮助产品/方案同学理解平台能力边界与接入方式，再导向现有正文。

## 目标
- 新增 `/docs-v2`，作为新的独立入口与信息架构层
- 优先服务产品/方案角色，30 秒内回答“平台能做什么”和“该怎么接”
- 保持现有正文 URL 不变，优先复用已有内容资产
- 将现有首页、使用指南、快速开始导读中的重复入口信息重组为更清晰的能力导览

## 非目标
- 不重写现有 `UI 接入`、`快速开始`、`API 参考`、`开放能力` 的深层正文
- 不迁移现有 API 页面到 `/docs-v2`
- 不替换当前 `Nextra` 文档框架
- 不新增搜索、CMS 或外部数据源

## 用户与优先级
本次 `/docs-v2` 优先服务：
- 产品经理
- 解决方案 / 售前
- 正在评估是否接入以及选择哪种接入方式的团队

相较之下，已经明确知道目标接口的开发者仍然可以直接进入原有正文页。

## 信息架构策略
`/docs-v2` 不承担正文树职责，只承担“入口层”和“分流层”职责。

新的一级结构采用能力地图优先方案：

1. `能力总览`
2. `接入方式`
3. `典型路径`
4. `文档入口`
5. `接口与资源`

这五块内容以单页分段方式呈现，用户在 `/docs-v2` 内先完成认知和选型，再通过 CTA 跳转到现有正文。

## 页面结构

### 1. Hero
- 明确说明平台覆盖的核心能力
- 强调三类主路径：`嵌入式 UI 工作台`、`服务端 PPT 生成链路`、`专项开放能力`
- 主 CTA 为“了解能力边界”和“选择接入方式”

### 2. 能力地图
将现有内容重组为四类能力卡：
- `内嵌工作台`
- `API 生成与管理`
- `开放能力`
- `模板 / PPT / 记录`

每张卡都承担“能力边界 + 对应正文入口”双重作用。

### 3. 接入方式选择
通过决策卡回答：
- 什么时候该用 `UI 接入`
- 什么时候该用 `API 接入`
- 什么时候该用 `开放能力`

这一段重点是“如何选”，不是“如何实现”。

### 4. 典型路径
沉淀三条高频阅读路径：
- 嵌入完整 AI PPT 工作台
- 从后端跑通 PPT 生成
- 只接 HTML 转 PPTX 等专项能力

每条路径都由一小段说明 + 顺序链接组成。

### 5. 文档入口总览
最后提供一个清晰的入口集合，集中跳转到：
- `/ui-integration`
- `/getting-started`
- `/api-reference`
- `/open-capabilities`
- `/pricing`

## 路由与主题策略
优先沿用当前 `Nextra` 站点能力，在 `content/docs-v2/` 下新增内容，并通过元信息将 `/docs-v2` 作为顶层独立入口暴露。

约束：
- `/docs-v2` 需要弱化或关闭默认文档侧栏
- 页面采用更强的 landing/overview 风格，但仍处于当前文档站视觉体系内
- 不影响原有文档路径的渲染与导航

## 风险与取舍

### 风险 1：入口页太像营销页
取舍：所有文案仍然围绕能力边界、接入方式和文档路径，不引入营销型转化内容。

### 风险 2：`/docs-v2` 与现有首页重复
取舍：保留现有首页稳定性，但让 `/docs-v2` 承担新的入口逻辑；后续可视情况再决定是否合并。

### 风险 3：Nextra 默认布局干扰独立入口感
取舍：优先通过页面级 theme 配置和局部样式实现弱侧栏 / 全宽布局，避免重写主题框架。

## 验收标准
- `/docs-v2` 可访问
- 页面能清楚表达平台能力边界与接入方式选择
- 页面中的所有主路径均链接到现有正文页
- 原有正文页路由和导航不受破坏
- `pnpm test` 和 `pnpm build` 通过
