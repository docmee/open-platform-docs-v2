# API Reference 文档统一改造设计

## 背景

当前 `content/api-reference` 下的接口详情页存在明显不一致：

- 一部分页面只有基础的接口信息和少量说明，缺少字段级参数表和响应表。
- 一部分页面已经包含示例代码，但请求示例、响应示例、字段说明、重点信息展示方式不统一。
- 大多数页面的路径仍为相对路径，未统一补全到完整的线上访问地址。
- `POST`、`GET` 等方法没有统一的视觉高亮，关键约束也缺少明显的强调样式。

本次改造目标是在不重构站点文档系统的前提下，统一 API 接口详情页的结构、信息密度和可读性。

## 目标

对 `content/api-reference` 下所有接口详情页统一完成以下改造：

1. 将接口方法使用不同颜色高亮展示，至少区分 `POST` 和 `GET`。
2. 将所有接口路径补全为以 `https://open.docmee.cn` 为基准的完整路径。
3. 为每个接口补充完整的请求参数说明，包括：
   - 最小可运行请求示例
   - 字段级表格说明
   - 必填项、类型、示例值、用途说明
4. 为每个接口补充响应示例与字段级响应表。
5. 对重要信息、关键约束、推荐做法进行粗体和高亮展示。
6. 新增一个支持点击复制的 `InlineCode` 组件，用于路径、字段名、Header、枚举值、短代码片段等行内代码展示。
7. 默认在接口页中提供 `cURL` 请求示例；已有 `Node.js` 示例的页面继续保留并统一格式。

不在本次范围内的内容：

- 不重做 `api-reference/index.mdx` 和 `api-reference/all-api.mdx` 的信息架构。
- 不将所有接口抽象成 JSON/YAML 驱动的数据系统。
- 不承诺补齐所有语言的 SDK 示例，只保证 `cURL` 为默认示例，保留或整理已有 `Node.js` 示例。

## 范围

处理范围为 `content/api-reference` 下所有具体接口详情页，不包含：

- `content/api-reference/index.mdx`
- `content/api-reference/all-api.mdx`

按当前仓库结构，涉及目录包括：

- `auth`
- `v2`
- `v1`
- `templates`
- `ppts`
- `custom-templates`
- `records`
- `portable`

## 方案概览

采用“方案 2：内容统一改造 + 增加少量通用样式/短组件”。

该方案保留现有 MDX 文档组织方式，不引入复杂的数据驱动层，而是在站点中新增轻量可复用组件和统一样式，并逐页重写接口内容结构。

### 为什么选这个方案

- 能直接满足本次批量改造目标，落地快，风险低。
- 能通过统一组件解决视觉一致性问题，避免在 40 多个页面里重复写高亮样式。
- 不改变站点路由和文档入口结构，避免影响现有导航与链接。
- 对后续新增接口页也有可复用价值。

## 组件与样式设计

### 1. `InlineCode`

新增通用行内代码组件，适用于：

- 接口路径
- Header 名
- 参数名
- 响应字段名
- 枚举值
- URL
- 短 token/key 示例

预期能力：

- 默认以比普通正文更明确的代码样式渲染。
- 支持点击复制。
- 复制成功后提供轻量反馈，例如“已复制”态或图标状态变化。
- 适合作为 MDX 中的基础表达组件，替代大量普通反引号写法。

设计约束：

- 交互应足够轻，不干扰阅读流。
- 对移动端可点击区域友好。
- 默认使用 ASCII 文本，不额外引入复杂依赖。

### 2. `HttpMethodBadge`

新增方法徽标组件，用于统一高亮接口方法。

要求：

- `POST` 和 `GET` 使用不同颜色。
- 在视觉上足够醒目，但不喧宾夺主。
- 支持后续扩展到其他方法，如 `PUT`、`DELETE`，即使本次未大量使用。

### 3. `ImportantNote` 或等价高亮提示块

用于承载：

- 前置条件
- 调用限制
- 字段取值约束
- 流式/异步/下载等特殊行为说明
- 推荐调用顺序

要求：

- 统一高亮背景和边框风格。
- 关键信息使用粗体。
- 避免每个页面手写不同样式的提示段落。

### 4. 表格与示例样式统一

对参数表、响应表、示例代码块做轻量统一，不追求重设计，只保证：

- 表格列语义统一。
- 字段名、类型、示例值的可读性更高。
- 代码块中的 URL、Header、关键字段与正文表达一致。

## 接口详情页统一版式

所有接口详情页统一调整为以下结构：

### 1. 接口信息

包含：

- 方法：使用 `HttpMethodBadge`
- 完整路径：统一为 `https://open.docmee.cn/...`
- 鉴权方式
- 请求体类型或参数位置

### 2. 接口作用

用 1 到 3 句解释该接口的用途、所属流程以及常见调用时机。

### 3. 重要信息

使用统一高亮提示块集中表达：

- 必要前置步骤
- 限制条件
- 推荐调用方式
- 兼容性说明
- 流式/异步/下载注意事项

### 4. 请求参数

统一拆为两层：

- 最小可运行请求示例
  - 默认提供 `cURL`
  - 已存在 `Node.js` 的页面继续保留并统一 URL 与字段
- 请求参数表
  - 至少包含：参数名、类型、必填、示例、说明

针对不同接口类型的约束：

- `GET`：明确 Query 参数与 Header
- `POST application/json`：按 JSON Body 字段组织
- `POST multipart/form-data`：区分表单字段与文件字段

### 5. 响应示例

提供标准成功响应示例。

如接口存在特殊响应形式，按实际情况补充：

- 流式响应片段
- 下载链接响应
- 异步状态响应

### 6. 响应字段说明

字段级表格至少包含：

- 字段名
- 类型
- 示例
- 说明

对嵌套字段使用展开写法，例如：

- `data.id`
- `data.list[]`
- `data.records[].pptId`

### 7. 多语言示例

处理规则：

- `cURL` 作为默认请求示例新增或补齐
- 已存在 `Node.js` 示例的页面继续保留
- 不强行给每个页面补齐四种语言，以避免信息膨胀和维护成本过高

## 内容补全策略

部分接口当前仓库中信息较少，本次补全遵循以下原则：

- 优先依据当前页面现有文案、同目录接口、相邻流程接口和示例代码推断字段含义。
- 对明显可以确定的字段，补充保守且准确的说明。
- 对无法从上下文明确确认的行为，不虚构过度细节。
- 对返回值结构采用“已知字段明确列出，未知部分用保守表述”的方式处理。

目标是提升可用性，同时避免生成看似完整但实际不可靠的接口文档。

## 实现步骤

### 步骤 1：确认站点组件接入点

检查当前 Nextra/MDX 组件注入方式，确定：

- MDX 自定义组件注册位置
- 文档样式落点
- 是否已有可复用的 callout/code 组件

### 步骤 2：实现轻量通用组件

新增并接入：

- `InlineCode`
- `HttpMethodBadge`
- `ImportantNote` 或等价提示块

必要时补充少量样式文件或主题覆盖。

### 步骤 3：先迁移一组代表性接口页

优先覆盖以下类型，验证模板可行性：

- `GET` 接口
- `POST JSON` 接口
- `POST multipart/form-data` 接口
- 流式响应接口
- 下载类接口

### 步骤 4：批量迁移所有接口详情页

按统一版式处理全部接口页面，重点保证：

- 完整路径统一
- 请求表示例齐全
- 响应示例和响应表齐全
- 重要信息高亮统一
- `InlineCode` 使用一致

### 步骤 5：全量检查与修正

对批量迁移后页面做抽查和构建验证，修正：

- 编译错误
- MDX 语法问题
- 漏掉的相对路径
- 表格列不一致
- 交互组件在移动端或深色主题下的可读性问题

## 验证策略

完成实现后至少执行以下验证：

1. 运行现有测试，确认未破坏仓库测试基线。
2. 运行构建或等价编译检查，确认新增 MDX 组件和所有接口页可正常编译。
3. 抽查典型页面：
   - `GET`
   - `POST JSON`
   - `POST multipart/form-data`
   - 流式接口
   - 下载接口
4. 检查 `https://open.docmee.cn` 是否已替换所有详情页中的相对路径。
5. 手工验证 `InlineCode` 的复制交互和 `HttpMethodBadge` 的颜色区分是否成立。

## 风险与应对

### 风险 1：部分接口字段语义不完整

应对：

- 使用保守描述。
- 优先覆盖最关键的调用字段和返回字段。
- 不臆造超出上下文的信息。

### 风险 2：批量改动导致 MDX 编译失败

应对：

- 先实现组件并验证接入。
- 逐批修改页面而不是一次性机械替换。
- 通过构建检查及时发现问题。

### 风险 3：页面篇幅显著变长

应对：

- 保证结构统一，但避免无意义冗长。
- 代码示例以 `cURL` 为默认，减少多语言重复。
- 重点信息放进高亮块，帮助快速扫描。

## 预期结果

改造完成后，`api-reference` 详情页应具备以下特征：

- 每页结构统一，便于阅读和维护。
- 方法、路径、字段、重点约束一眼可识别。
- 用户可直接复制路径、字段名、Header 等关键行内代码。
- 每个接口都具备至少一份可运行请求示例和一份明确响应示例。
- 参数和响应不再只停留在示例层面，而是具备字段级说明能力。
