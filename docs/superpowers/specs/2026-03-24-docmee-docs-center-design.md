# DocMee 文档中心视觉重设计

## 背景
当前仓库已经是一个基于 `Next.js 15`、`Nextra 4` 和 `nextra-theme-docs` 的文档站，具备稳定的路由、内容目录和 API 文档结构，但整体视觉仍接近默认主题，缺少产品化文档站应有的品牌感、层次感和高级感。

用户希望将当前文档站改造成接近 `https://langfuse.com/docs` 的风格，同时保留 DocMee 自身品牌，不直接照搬素材，不重做信息架构。

## 目标
- 在不推翻现有文档结构的前提下，显著提升站点视觉质量
- 将整体体验拉近到 `langfuse docs` 这一类 SaaS 产品文档风格
- 保留现有 `Nextra` 文档框架能力，包括侧栏、目录、正文、MDX 渲染
- 使用“克制的科技冷紫”作为主强调色，建立统一视觉系统
- 改善首页、导航、正文、代码块、表格和提示块的细节表现

## 非目标
- 不重做文档信息架构
- 不将站点改造成营销官网
- 不大规模重写现有文档内容
- 不引入新的 CMS、搜索服务或数据源
- 不追求对 `langfuse` 的逐像素复刻

## 设计范围
本次改版限定在表现层和少量布局层：

- `app/layout.tsx`
- `app/global.css`
- 必要时新增少量首页或 MDX 展示组件
- 必要时微调 `content/index.mdx` 的首页区块结构

以下内容默认保持不变：

- 文档路由
- 左侧导航信息架构
- 主要 MDX 内容页面
- API 参考的目录组织

## 设计原则

### 1. 高相似借鉴，不直接复制
目标是让用户在第一眼感受到与 `langfuse docs` 同类的文档产品气质，但不直接复制其品牌元素、文案、图形或颜色。

### 2. 文档优先，不官网化
首页和内页都要强化“文档入口”和“阅读效率”，而不是转向营销型落地页。

### 3. 克制的紫罗兰体系
强调色采用偏冷、低饱和的紫罗兰色调，而非高饱和、模板化的“紫色 SaaS”风格。

### 4. 页面层次来自边界、留白和节奏
核心质感依赖于：
- 浅暖灰背景
- 细描边
- 大圆角
- 低对比阴影
- 稍强的排版层级

而不是依赖高饱和背景色块或过重装饰。

## 视觉系统

### 色彩
- 页面主背景：浅暖灰，而非纯白
- 内容容器背景：接近白色，但与页面背景保留轻微层差
- 主强调色：科技冷紫，用于按钮、链接、选中态、焦点态
- 辅助强调层：雾紫或极淡紫灰，用于光晕、hover 和局部背景层
- 正文字色：中性炭灰，保证可读性

约束：
- 不允许大面积高饱和紫色铺底
- 不让正文文字或图标长期处于紫色主导状态
- 紫色仅承担强调，而非替代中性色体系

### 形态
- 卡片、按钮、提示块、代码块统一更大的圆角
- 使用细边框而不是重投影
- 阴影存在但弱化，以“浮起一点”为上限

### 背景氛围
- 首页允许加入极淡的渐变、光晕或网格感背景
- 内页氛围必须弱于首页，避免影响阅读
- 背景效果应服务于“产品感”，不制造视觉噪音

## 首页设计
首页继续作为文档入口，不改成独立官网。推荐拆分为 4 个模块：

### 1. Hero
保留文档首页定位，使用更强的标题层级和更高质量的版式。

包含：
- 一个明确的主标题
- 一段简短的产品型说明文案
- 一个主 CTA，进入“快速开始”
- 一个次 CTA，进入“API 参考”或“开发指南”

视觉要求：
- 使用轻量紫罗兰光晕或渐变背景
- 标题更紧凑，更接近产品文档站而非普通博客
- 不放过多营销信息

### 2. 核心入口卡片
保留现有关键入口，但改为更有秩序的卡片模块，例如：
- 产品介绍
- 快速开始
- 开发指南
- API 参考
- 更新日志

每张卡片包含：
- 标题
- 一句说明
- 轻量图标、编号或标签

目标：
- 让首页看起来像文档中枢，而不是链接列表

### 3. 接入流程
将接入流程改为结构化步骤区，而非普通段落堆叠。

可选表现：
- 横向步骤卡片
- 序号流程条

目标：
- 让首次访问用户快速理解接入节奏

### 4. 推荐阅读 / 常见路径
使用任务导向入口，而非再次重复导航，例如：
- 第一次接入先看什么
- 想直接调接口先看什么
- 想自定义模板先看什么

目标：
- 帮助用户按任务而不是按栏目进入内容

## 文档内页设计

### 顶部导航
- 从默认文档条改成更轻、更细、更接近透明浮层的顶部区域
- 与背景融为一体，但保留足够的可识别性
- 保留站点 Logo 和基础导航能力

### 左侧侧栏
- 保留现有 Nextra 目录结构
- 改善间距、字重和选中态
- 选中项使用柔和紫色高亮、细竖条或轻底色，不使用厚重色块

### 中间正文
- 提升阅读区宽度控制
- 放大标题层级差异
- 增加段落、列表、分割线之间的节奏感
- 让正文从默认主题观感转为更成熟的产品文档排版

### 右侧 TOC
- 保留，但视觉弱化
- 突出当前锚点而不抢正文注意力

## 组件级样式目标

### 按钮
- 主按钮使用冷紫强调
- 次按钮使用描边或低对比填充
- hover 和 focus 需清晰但克制

### 卡片
- 统一圆角、边框和轻微浮层感
- hover 不应夸张，只做轻微位移或背景变化

### 代码块
- 相比默认主题更像“文档产品代码卡片”
- 拥有清晰边界、适当标题区和更稳定的滚动体验
- 配色与整体冷紫体系协调，但不能牺牲可读性

### 表格
- 增强网格清晰度
- 表头、行间、圆角和边框风格与卡片系统一致

### 提示块 / 引用块
- 使用轻背景和边框强调信息层次
- 不采用过于艳丽的语义色

### 行内代码 / 链接
- 行内代码需要更清楚的底色和边界
- 链接使用冷紫强调，但 hover 时避免过亮

## 技术方案
继续使用现有技术栈：
- `Next.js 15`
- `Nextra 4`
- `nextra-theme-docs`
- `MDX`
- `Tailwind CSS 4`

实现策略：
- 优先通过 `app/layout.tsx` 配置 `Navbar`、`Layout`、`Footer` 和站点外层容器
- 通过 `app/global.css` 重写全站视觉变量、容器、排版和主题细节
- 必要时在首页 MDX 中引入少量结构化组件，但避免引入复杂前端状态
- 尽量复用 Nextra 内置结构，不自建完整文档主题

## 实施边界
- 不替换现有文档框架
- 不重写目录生成机制
- 不侵入每个文档页单独写样式，优先使用全局规则和少量通用组件
- 如果某些 Nextra 默认 DOM 结构限制了 100% 复刻，则优先保留稳定性和维护性

## 风险与取舍

### 风险 1：过度追求相似导致维护成本升高
取舍：优先实现“第一眼相似”和“整体质感相似”，不追求主题内部完全重建。

### 风险 2：紫罗兰主题过重影响技术文档可读性
取舍：紫色仅用于强调层，正文仍以中性色主导。

### 风险 3：首页精致但内页回落到默认主题
取舍：内页导航、正文、代码块和表格必须一并改造，保证整体一致性。

## 验收标准
- 首页在视觉上明显脱离默认 Nextra 主题
- 首页具备 `Hero`、核心入口卡片、接入流程、推荐阅读四个模块
- 整站主强调色为克制的科技冷紫，而非高饱和模板紫
- 顶部导航、左侧侧栏、正文、右侧目录形成统一视觉语言
- 代码块、表格、提示块和链接样式完成统一升级
- 保持现有主要文档路由和内容结构可用
- `pnpm build` 成功

## 后续扩展方向
- 再补充一套更完整的首页插图或图形系统
- 为 API 页面增加统一的接口说明模块组件
- 根据真实品牌规范微调冷紫配色和 Logo 呈现
